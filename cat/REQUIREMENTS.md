# Zig cat 구현 요구사항

이 프로젝트는 Unix 계열 도구인 `cat`을 Zig로 다시 구현하면서 Zig의 기본 문법, 표준 라이브러리, 파일 I/O, 에러 처리, 테스트 작성 방식을 익히는 것을 목표로 한다. GNU coreutils의 `cat`을 완전히 대체하는 것이 목적이 아니며, 학습 난이도를 단계적으로 높일 수 있는 범위로 구현한다.

## 목표

- 명령행 인자를 파싱해 하나 이상의 파일 내용을 표준 출력으로 복사한다.
- 파일 인자가 없으면 표준 입력을 읽어 표준 출력으로 복사한다.
- `-` 인자는 표준 입력을 의미한다.
- 여러 입력이 주어지면 입력 순서대로 이어서 출력한다.
- 정상 출력은 `stdout`에만 기록하고, 오류 메시지는 `stderr`에 기록한다.
- Zig의 `std.fs`, `std.Io`, allocator, error union, `defer`, 테스트 기능을 직접 사용해본다.

## 비목표

- GNU/BSD `cat`의 모든 옵션을 동일하게 구현하지 않는다.
- 인코딩 변환, 줄 끝 변환, 파일 감시, 병렬 처리, 대용량 최적화까지 다루지 않는다.
- OS별 특수 파일, 권한, 터미널 제어 문자의 모든 차이를 완벽히 흉내 내지 않는다.

## 기본 동작 요구사항

### 입력 처리

- `cat file.txt`는 `file.txt`의 바이트를 그대로 `stdout`에 출력해야 한다.
- `cat file1 file2`는 `file1` 다음에 `file2`를 이어서 출력해야 한다.
- `cat`처럼 파일 인자가 없을 때는 `stdin`을 EOF까지 읽어 `stdout`에 출력해야 한다.
- `cat -`는 `stdin`을 읽어 `stdout`에 출력해야 한다.
- `cat a - b`는 `a`, `stdin`, `b` 순서로 출력해야 한다.
- 입력 데이터는 텍스트로 해석하지 않고 바이트 스트림으로 취급한다.

### 오류 처리

- 존재하지 않는 파일을 열면 프로그램은 실패 상태로 종료해야 한다.
- 읽을 수 없는 파일을 만나면 오류 메시지를 `stderr`에 출력해야 한다.
- 여러 파일 중 하나에서 오류가 발생해도 가능한 한 다음 입력 처리를 계속한다.
- 하나 이상의 입력에서 오류가 발생했다면 최종 종료 코드는 실패여야 한다.
- 오류 메시지에는 프로그램 이름, 대상 경로, 실패 원인을 포함한다.

### 출력 처리

- 정상 데이터는 `stdout`에만 출력한다.
- 디버그 로그, 안내 문구, 테스트 메시지는 `stdout`에 출력하지 않는다.
- 출력 전후로 임의의 개행을 추가하지 않는다.
- 입력 파일의 내용을 바이트 단위로 보존한다.

## 옵션 요구사항

학습 단계에 맞춰 아래 옵션을 순서대로 구현한다.

### 1단계: 옵션 없음

- 파일, `stdin`, `-` 입력만 지원한다.
- 잘못된 옵션은 파일명으로 해석하지 않고 오류로 처리한다.

### 2단계: 줄 번호

- `-n`, `--number`를 지원한다.
- 모든 출력 줄 앞에 1부터 시작하는 줄 번호를 붙인다.
- 여러 파일을 이어서 출력할 때 줄 번호는 초기화하지 않고 계속 증가한다.

### 3단계: 빈 줄 제외 줄 번호

- `-b`, `--number-nonblank`를 지원한다.
- 빈 줄에는 번호를 붙이지 않는다.
- `-b`와 `-n`이 함께 주어지면 `-b`가 우선한다.

### 4단계: 도움말

- `-h`, `--help`를 지원한다.
- 사용법과 지원 옵션을 `stdout`에 출력하고 성공 상태로 종료한다.

## 명령행 파싱 요구사항

- 옵션은 파일 인자보다 앞에 있거나 섞여 있어도 인식한다.
- `--` 이후의 인자는 옵션으로 해석하지 않고 파일명으로 처리한다.
- 알 수 없는 옵션은 `stderr`에 오류를 출력하고 실패 상태로 종료한다.
- 옵션 파싱 로직은 출력 로직과 분리해 테스트 가능하게 만든다.

## 구조 요구사항

- `src/main.zig`는 CLI 진입점, 인자 처리, 종료 코드 처리를 담당한다.
- `src/root.zig` 또는 별도 모듈은 실제 `cat` 동작을 담당한다.
- 핵심 복사 함수는 입력 reader와 출력 writer를 받아 동작하게 만들어 파일 시스템 없이 테스트할 수 있어야 한다.
- 전역 상태를 피하고 필요한 상태는 구조체나 함수 인자로 전달한다.
- 큰 파일을 한 번에 메모리에 올리지 말고 고정 크기 버퍼로 반복 복사한다.

## 테스트 요구사항

- `zig build test`로 모든 테스트가 실행되어야 한다.
- 메모리 누수가 없어야 한다.
- 최소한 아래 사례를 테스트한다.

1. 단일 입력을 그대로 출력한다.
2. 여러 입력을 순서대로 이어 붙인다.
3. 빈 입력을 정상 처리한다.
4. `-n`이 모든 줄에 번호를 붙인다.
5. `-b`가 빈 줄을 제외하고 번호를 붙인다.
6. `-b`가 `-n`보다 우선한다.
7. 알 수 없는 옵션을 오류로 처리한다.
8. `--` 이후의 값을 파일명으로 취급한다.

## 구현 단계 제안

1. 기본 프로젝트 템플릿의 예제 출력과 예제 테스트를 제거한다.
2. `copy(reader, writer)` 함수를 작성해 바이트 스트림 복사를 구현한다.
3. 파일 인자가 없을 때 `stdin`에서 읽도록 연결한다.
4. 파일 경로 목록을 순회하며 순서대로 출력한다.
5. 실패한 파일이 있어도 나머지 파일을 처리하고 마지막에 실패 종료 코드를 반환한다.
6. 옵션 파서와 설정 구조체를 추가한다.
7. 줄 번호 출력 기능을 별도 함수로 구현한다.
8. 통합 테스트나 실행 예제를 추가해 실제 CLI 동작을 확인한다.

## 완료 기준

- `zig build test`가 성공한다.
- `zig build run -- sample.txt`가 `sample.txt`의 내용을 그대로 출력한다.
- `zig build run -- -n sample.txt`가 줄 번호를 출력한다.
- 오류와 정상 출력이 `stderr`와 `stdout`으로 분리된다.
- 핵심 로직이 파일 시스템에 직접 의존하지 않아 단위 테스트로 검증 가능하다.
